Klara Thelusma, Maya Anderson, Pedro Gallardo Nascimento, Colin Nguyen

Introduction¶

With an increasingly interconnected world, some movies exist in multiple languages so that they could be distributed in different nations across the world. However, with being shown in various countries with unique cultures and varying values, a question arises: does the language and region that a movie is in affect its reception? Using the IMDB dataset that includes thousands of movies and their data, our group wants to answer this question by analyzing variables such as language, region, and genre. In the contemporary use of the data, we must acknowledge that the predictions and patterns that we may find aren’t all-encompassing or consistently accurate; as we know, society has changed over time, and so the findings we uncover in this age may not be wholly applicable to a past time, or even the future.

Database Design¶

The database is structured by using titleId to bridge information across different tables. The database also includes many copies of the same movies, with the subtle differences being in the differences of the language that the specific version is marketed in, or the original title. We will be paying close attention to these copies in different languages as we want to see if the language and region that a movie is released in affects its ratings. The information regarding region is also important to see if a genre performs the same across the world, or if certain regions prefer one genre over the other. This approach allows us to easily compare the reception of certain movies and their genres in different regions as there are multiple copies of the same movie to compare each other to, rather than having the confounding variable of the ratings being affected by the movie itself and not its genre.

WITH MaxRatings AS (
    SELECT 
        title_akas.titleId, 
        title_akas.region, 
        ratings.averageRating
    FROM 
        title_akas
    JOIN 
        ratings
    ON 
        title_akas.titleId = ratings.titleId
    WHERE 
        title_akas.region IN {English_region}
    ORDER BY 
        ratings.averageRating DESC
    LIMIT 100
)
SELECT 
    genres.genre, 
    COUNT(*) AS genre_count
FROM 
    MaxRatings
JOIN 
    genres
ON 
    MaxRatings.titleId = genres.titleId

    genres.genre
ORDER BY 
    genre_count DESC
"""

For instance, in the get_genre_distribution(region_filter, rating_type) function, rather than writing separate functions to extract the minimum and maximum ratings, we consolidated this logic into one function. This approach streamlined our code, but we were mindful of balancing brevity with readability. While reducing code length can often improve clarity, overly complex one-liners can become difficult to follow. To address this, we focused on organizing our code with clear structures and logical flows. We used consistent naming conventions, broke down larger functions into smaller, well-defined components, and ensured that each part of the code was easy to understand at a glance. Overall, our goal was to create clean, readable, and efficient code that minimizes redundancy without sacrificing clarity, ensuring it remains accessible to anyone working with it now or in the future.

Another benefit is that code that is reversible often tends to be more readable, since code that is able to be easily changed is more readable than code that is not reversible.
